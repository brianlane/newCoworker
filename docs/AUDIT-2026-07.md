# Full Application Audit — July 17, 2026

Read-only audit of the New Coworker monorepo: security, dependencies, config
hygiene, and code quality across the Next.js app (`src/`, ~176k LOC), Supabase
edge functions (`supabase/functions/`, ~35k LOC), VPS services (`vps/`, ~20k
LOC), CI workflows, and operational scripts. A core goal was verifying the
buyer-facing security claims in `README.md` against the code that ships.

## Executive summary

**The posture is strong and the README does not overstate it.** No critical or
high-severity platform findings. Every one of the ~200 Next.js API routes and
33 edge functions has an appropriate auth gate; webhook signatures (Stripe,
Telnyx Ed25519, Meta HMAC, Vagaro per-tenant token) are verified before
processing; the per-tenant gateway-token binding works exactly as documented;
secret comparison is timing-safe everywhere sampled; stored secrets use the
AES-256-GCM envelope with fail-closed reads. The objective baseline is
excellent: **0 lint errors, 0 type errors, 10,325/10,325 tests passing, 100%
line/branch/function coverage on the gated scope, 0 npm-audit vulnerabilities
in the app's production tree, and zero drift between `.env.example` (125 vars)
and the env vars actually read in `src/`.**

The findings below are one real (dev-tooling) dependency vulnerability, a
handful of deliberate availability-over-security tradeoffs worth tracking as
accepted risks, and low-severity hygiene items.

Severity scale: Critical / High / Medium / Low / Informational.

## Medium

### M1. `zapier/` has 2 high-severity advisories via `zapier-platform-cli` (`adm-zip < 0.6.0`)

`npm audit` in `zapier/` reports GHSA-xcpc-8h2w-3j85 (crafted ZIP triggers a
4 GB memory allocation) through `zapier-platform-cli`'s `adm-zip` dependency.
**No fix available upstream yet.**

- Impact: limited — this is the developer/CI tool used to build and push the
  Zapier integration, not tenant-facing runtime code. Exploitation requires
  feeding the CLI a malicious ZIP.
- Recommendation: track `zapier-platform-cli` releases and bump when a fixed
  version ships; consider an `overrides` pin to `adm-zip >= 0.6.0` in
  `zapier/package.json` if the CLI tolerates it. Note this tree is *outside*
  the root `Dependency Audit` CI job (`.github/workflows/audit.yml` only
  audits the root lockfile), so this class of finding is currently invisible
  to CI — see M2.

### M2. CI dependency audit only covers the root package tree

`.github/workflows/audit.yml` and the CI `security` job run `npm audit` on
the root lockfile only. The independent trees — `vps/chat-worker`,
`vps/voice-bridge`, `vps/data-api`, `vps/aiflow-render`,
`cloudflare/email-worker`, `zapier/` — are never audited in CI (all currently
clean except `zapier/`, verified manually in this audit).

- Recommendation: extend the audit workflow with a matrix step over each
  directory containing a `package-lock.json`. The VPS trees matter most: they
  run on tenant boxes.

### M3. In-memory rate limiting is the primary limiter on serverless — REMEDIATED

> **Remediated (PR: audit M3/L3 fixes):** `rateLimitDurable` now guards
> `/api/onboard/check-email`, `/api/onboard/set-password`,
> `/api/onboard/finalize-signup`, `/api/onboard/draft` (POST + GET),
> `/api/widget/session`, and the per-IP layer of `/api/widget/message`.
> Original finding kept below for the record.

`src/proxy.ts` and most route-level `rateLimit(...)` calls use the per-process
`Map` limiter (`src/lib/rate-limit.ts`), which on Vercel is per-isolate and
ephemeral — the code's own docblock says it "barely binds across the fleet."
The durable Postgres-backed `rateLimitDurable` is used on only five routes
(`/api/onboard/chat`, `/api/onboard/website-preview`, `/api/contact`, the two
AiFlow library LLM routes).

- Impact: unauthenticated, non-LLM endpoints (e.g. `/api/onboard/check-email`,
  which by design leaks account existence; `/api/widget/session`) get only
  per-isolate limits, so a distributed or isolate-spreading caller has a much
  larger effective quota than the configured numbers suggest.
- Recommendation: move account-enumeration-adjacent and abuse-prone
  unauthenticated endpoints to `rateLimitDurable` (its 1.5 s bounded RPC +
  in-memory fallback already handles the availability concern), or add edge
  (Cloudflare/Vercel WAF) limits in front.

## Low

### L1. Documented fail-open windows in gateway-token verification

Two deliberate availability-over-security tradeoffs, both clearly commented,
worth tracking as accepted risks:

- `resolveRowboatWebhookClaims` (`src/lib/rowboat/webhook-jwt.ts`): a DB error
  during per-tenant token resolution falls back to accepting the **shared**
  secret even for tenants whose per-tenant token is CONFIRMED. During a DB
  blip, the cross-tenant JWT-forgery protection (the whole point of
  exclusivity) is temporarily suspended.
- `verifyGatewayTokenForBusiness` (`src/lib/rowboat/gateway-token.ts`): same
  fail-open shape on the bearer path (lower stakes — an unknown token still
  has to equal the shared secret).

Recommendation: emit a `telemetry_events` row when the fail-open path is
actually taken, so a sustained DB degradation that silently widens the trust
boundary is visible in ops dashboards rather than only in code comments.

### L2. Dependabot auto-merge is enabled for MAJOR version bumps

`.github/workflows/dependabot-automerge-label.yml` labels **all** update types
(patch, minor, and major) as `dependabot-automerge`; the merger's only gate is
"every check green + no unresolved threads." The merger script's skip message
("lacks label — likely a major update") no longer matches the labeler's actual
behavior. CI is thorough (build, 10k tests, e2e with live model, worker
integration), so this is a considered choice — but a major bump of e.g.
`stripe` or `@supabase/ssr` can land with zero human eyes if nothing breaks a
test.

- Recommendation: either exclude `version-update:semver-major` from the label
  step (making majors require a human merge), or update the stale comment so
  the policy is honestly documented.

### L3. `POST /api/onboard/draft` allows unauthenticated first-claim of a draft slot — REMEDIATED

> **Remediated (PR: audit M3/L3 fixes):** first-claiming a draft slot whose
> `businesses` row already exists now requires the HMAC `onboardingToken`
> minted by `/api/business/create` (the pre-create first save with a fresh
> client-side UUID stays open — that id has never left the browser). The
> questionnaire client sends the token on both draft POSTs. Original
> finding kept below for the record.

The draft store is token-paired (first write binds a `draftToken`; later
writes/reads must present it), but the **first** write for a given
`businessId` is open. An attacker who learns a fresh businessId (it appears in
client-side onboarding state) could pre-claim the draft slot and cause the
legitimate client's saves to 403 (nuisance/DoS on onboarding recovery; no data
disclosure — reads require the matching token).

- Recommendation: bind draft creation to the `onboardingToken` minted by
  `/api/business/create` (the same gate `/api/checkout` uses).

### L4. Meta webhook GET handshake uses non-timing-safe compare

`src/app/api/webhooks/meta/route.ts` compares `hub.verify_token` with `!==`.
Exposure is minimal (the token only gates the one-time subscription handshake;
POSTs are HMAC-verified), but every other shared-secret compare in the repo is
timing-safe — this one is the outlier. One-line fix via
`timingSafeEqualUtf8`.

### L5. Full script/style CSP still deferred

`next.config.ts` ships a solid baseline (HSTS w/ preload, nosniff,
frame-ancestors 'none', base-uri, object-src, Permissions-Policy) but no
`script-src`/`style-src` policy — explicitly deferred pending a Report-Only
bake. XSS surface today is small (React escaping everywhere; the two
`dangerouslySetInnerHTML` sites are safe: JSON-LD with `<` escaping, and email
HTML through DOMPurify into a sandboxed iframe srcdoc). Keep the deferral on
the roadmap; a CSP is the backstop for the *next* injection bug, not the
current ones.

### L6. SSH trust-on-first-use window (documented)

Host-key pinning is strict after capture, but the very first connection to a
fresh image is TOFU, and re-image flows clear the pin
(`src/lib/hostinger/ssh-pinned.ts`; README states this honestly). Accepted
risk; a future hardening could fetch the host key out-of-band via the
Hostinger API if they expose it.

### I2. Route auth map — no unguarded mutating surface found

All ~200 API routes were mapped to their guards. Every route resolves to one
of: `requireAdmin` (43 admin routes), `requireBusinessRole` via the central
policy matrix (dashboard/aiflows/integrations), `assertCronAuth` (internal
sweeps), gateway-token binding (VPS/voice tools), webhook signature
verification, capability tokens (public docs/sign links: 256-bit, sha256
stored, uniform 404s), public API keys (`/api/public/v1/*`), widget
session bearers, or Stripe-session possession for post-checkout onboarding
(`finalize-signup` / `set-password` — the latter is create-only with an
explicit account-takeover analysis in its docblock; `/api/checkout` holds the
matching strict uniqueness gate that fails closed on lookup errors).

The MCP server (`/api/mcp`) verifies Supabase OAuth bearers via `getClaims`
and role-checks per business through the same `authz/policy.ts` matrix as the
dashboard — as the README claims.

### I5. Lockstep copies verified in sync

`deriveMonthlyQuotaWindow` (`supabase/functions/_shared/billing_period_window.ts`)
vs. the inline copies `deriveMonthlyQuotaWindowStart` in
`vps/chat-worker/worker.mjs` and `vps/voice-bridge/src/index.ts`: the
`addUtcMonthsClamped` helpers are byte-identical and the window derivation is
semantically identical (the VPS copies return only `startIso`, which is all
they consume). No drift.

### I6. CI/CD pipeline observations

- The pipeline is unusually rigorous: lint + build, typecheck (with
  `next typegen`), full test suite, Deno typecheck of edge functions, npm
  audit, worker integration against a real local Supabase, migration-drift
  check at PR time, gated Vercel deploy, then a live-model e2e suite — with
  Supabase migrations + edge-function deploy ordered *before* the production
  app deploy on main.
- `vercel.json` disables git auto-deploys so nothing ships without the gate.
- Workflow permissions are least-privilege (`contents: read` default);
  the automerge workflow uses `pull_request_target` for labeling only, with
  no checkout of PR code (no injection surface).
- Supabase CLI is version-pinned with a comment explaining why. Actions are
  pinned to major tags (`@v7`), not SHAs — acceptable, but SHA-pinning would
  harden against tag-moving supply-chain attacks.
